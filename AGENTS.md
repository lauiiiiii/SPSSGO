# AI 编程协作规则

这是本项目的 AI coding agent 规则入口。改代码前先读相关实现，按这里的风格来，不要把项目改成模板味。

## 基本原则

- 先读代码，再改代码；先理解目录、调用链、命名和现有写法。
- 只改当前任务相关内容，不做顺手重构，不批量格式化无关文件。
- 优先沿用已有模块、service、repository、composable、utils、样式和测试写法。
- 命名要直白、贴近业务；不要为了显得高级增加抽象层或复杂设计。
- 不确定业务规则时先说明假设，别擅自编规则。

## 注释规范

- 代码自解释优先；只注释隐藏逻辑、业务特殊规则、历史兼容、边界限制、算法思路和风险点。
- 注释写给未来干活的人，不写给机器；只讲“为什么这样写”“有什么限制”“不能怎么改”。
- 语气极简、直接、工程化；可以有判断和提醒，但禁止情绪化吐槽、骂人、小作文。
- 禁止废话注释：不要写“定义变量”“调用接口”“循环遍历列表”这种复述代码的内容。
- 代码改了注释也要跟着改；临时兼容、临时补丁必须写背景和预计移除条件。

文件级注释：

- 新增核心文件顶部必须写 1 到 3 行，说明文件职责、业务范围、所在层级和限制。
- 大改旧文件时，如果顶部没有职责说明，顺手补；小修小补不要为了补注释扩大改动。
- 配置文件、纯数据文件、生成文件可以不加。
- 示例：`# 这里只放数据集版本的业务编排，别把路由和 SQL 细节塞进来`

函数 / 接口注释：

- 核心函数、公共方法、接口、算法逻辑要写文档注释，包含功能、入参、出参、异常或注意事项。
- 普通小函数不用硬写文档注释，名字能说明白就别废话。
- Python 用 docstring；JS/TS/Vue 复杂方法用 JSDoc，风格保持简短。

行内注释：

- 只给非常规写法、兜底逻辑、防报错、脏数据兼容、风控限制、魔法值写短注释。
- 常量、固定枚举、特殊配置必须说明业务含义，例如：`// 1=正常 2=封禁 3=临时冻结`
- 一段代码最多 1 到 2 条关键注释，宁少但狠。

优先这种味：

```python
# 这个地方有坑，别随便改
# 这里动了很容易炸
# 兼容历史脏数据，不可删除
# 这里先写死，后面再说
# 这个值不是最优，是目前能跑的
# 这里没必要抽象，复杂了反而出问题
# 失败就降级，别卡主流程
# 这里必须兜底，不然后面全挂
# 别动这个，动了后面一堆要改
# 这个逻辑是配合后面用的，别拆
```

一句话：不解释代码，专门提醒坑位，讲清后果，带点“别乱动”的压迫感。

## 解耦与分层

- 保持职责边界：route 管请求响应，service 管业务编排，repository 管数据访问，storage 管文件存储。
- 前端组件管展示和交互，复杂状态和流程放 composable，通用能力放 utils。
- 不把业务逻辑塞进路由、组件、脚本入口或超大函数。
- 新函数职责单一、名字说人话、方便测试。
- 拆分是为了解决真实复杂度，不是为了制造更多文件。

## 项目禁区

- 不确认不要删除 `tests/`、`scripts/`、`loadtests/`、`monitoring/`、`ops/`、`acceptance_refs/`。
- 不确认不要改环境文件、密钥配置、部署端口、数据库迁移和生产化配置。
- 不要覆盖用户未提交改动；工作区是脏的也只处理当前任务相关文件。
- 不要为了让测试通过降低校验、吞异常或删除断言。
- 无法运行测试时，最终说明里讲清原因和风险。
